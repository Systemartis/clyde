### Token usage location

Usage is **per-assistant-event** inside `message.usage`. Key fields:
- `input_tokens` — tokens in this request
- `output_tokens` — tokens generated
- `cache_creation_input_tokens` — tokens written to cache
- `cache_read_input_tokens` — tokens read from cache
- `cache_creation.ephemeral_1h_input_tokens` / `ephemeral_5m_input_tokens` — cache tier breakdown

Cost must be computed by the consumer using Anthropic's pricing table (model + token counts). No pre-computed cost field exists in the JSONL.

### File-touching tool patterns

All file-touching tools appear as `tool_use` items inside assistant `content` array:

| Tool | Key input fields |
|------|-----------------|
| `Write` | `file_path`, `content` |
| `Edit` | `file_path`, `old_string`, `new_string` |
| `MultiEdit` | `file_path`, `edits: [{old_string, new_string}]` |
| `Read` | `file_path`, optional `offset`, `limit` |
| `Bash` | `command`, `description` |

The `file_path` field is always absolute. Pattern: to detect file changes, scan `assistant` events for `tool_use` items where `name` in `{Write, Edit, MultiEdit}`.

### Sub-agent / Task indication

Sub-agents are represented as `queue-operation` events (not `isSidechain`):

```json
{
  "type": "queue-operation",
  "operation": "enqueue",
  "timestamp": "2026-04-28T09:53:39.405Z",
  "sessionId": "be28df01-...",
  "content": "<task-notification>...<status>completed</status><summary>Agent X completed</summary>...</task-notification>"
}
```

Operations: `enqueue`, `dequeue`, `remove`. The `content` field is XML-like with task-id, tool-use-id, output-file path, status, and summary.

The spawning `Agent` or `TaskCreate` tool_use is in the parent session's `assistant` events. Sub-agent spawning tools observed: `Agent`, `TaskCreate`, `TaskUpdate`.

- No distinct `summary` event type exists in any sampled session.
- Compaction state is not directly observable from the JSONL transcript alone.
- The `file-history-snapshot` event records tracked file backup state (not JSONL compaction).

## Bubble Tea v2 patterns

> Sourced from Context7 docs (charmbracelet/bubbletea). Confidence: HIGH for Cmd/Batch/Tick patterns; MEDIUM for v2-specific API changes.

### Model-View-Update (MVU) core

```go
type Model interface {
    Init() Cmd
    Update(Msg) (Model, Cmd)
    View() string
}
```

### Multi-pane layout (lipgloss)

No dedicated "pane manager" in Bubble Tea itself. The standard pattern is:

1. **Nested models** — each pane is its own `tea.Model` struct. The root model holds child models and delegates `Update` calls.
2. **Lipgloss joining** — `lipgloss.JoinHorizontal(lipgloss.Top, leftPane, rightPane)` and `lipgloss.JoinVertical(lipgloss.Left, topRow, bottomRow)` compose rendered strings in `View()`.
3. **Focus tracking** — a `focusedPane int` field on the root model; `FocusMsg` / `BlurMsg` from `tea.WithReportFocus()` option inform terminal-level focus.

### Background streaming (goroutines → tea.Cmd)

The canonical Bubble Tea pattern for external event sources (fsnotify, polling a file):

```go
// Return a Cmd from Init/Update that polls/watches for changes
func waitForFileChange(ch <-chan struct{}) tea.Cmd {
    return func() tea.Msg {
        return FileChangedMsg(<-ch) // blocks until signal
    }
}

// In Update: re-issue the Cmd after each message to keep watching
case FileChangedMsg:
    return m, waitForFileChange(m.fileCh)
```

For time-based polling (e.g. tail JSONL every N ms):
```go
func tickPoll() tea.Cmd {
    return tea.Tick(500*time.Millisecond, func(t time.Time) tea.Msg {
        return PollMsg(t)
    })
}
```

`tea.Batch(cmd1, cmd2)` runs multiple commands concurrently. `tea.Sequence(cmd1, cmd2)` runs them in order.

### Key caveat for fsnotify

`fsnotify.Watcher` must live in the **adapter layer** (or be passed as a channel to a tea.Cmd). The watcher goroutine sends events into a `chan tea.Msg`; the Cmd blocks on that channel. This is the standard bridge pattern — the domain and application layers never see `fsnotify`.

### Focus reporting across panes

`tea.WithReportFocus()` program option enables `FocusMsg` / `BlurMsg` messages. NOTE: requires terminal/multiplexer support. tmux needs `set -g focus-events on`. Ghostty supports it natively.

## Approach comparison

### Option A: Walking Skeleton (recommended)

**What:** Smallest possible end-to-end vertical slice. Read the project's JSONL files → decode last 5 events → render in a single Bubble Tea pane. All hexagonal layers present but thin.

**Deliverables for bootstrap-skeleton:**
- `go.mod` with `github.com/clyde-tui/clyde`, Go 1.23 directive
- Layer directories: `internal/domain/`, `internal/application/`, `internal/ports/`, `internal/adapters/jsonl/`, `internal/adapters/tui/`, `cmd/clyde/`
- Domain types: `Session`, `Event` (minimal — only fields needed for rendering)
- Port: `SessionSource` interface
- Adapter: `JsonlSessionSource` — reads the JSONL file for the current project's cwd, decodes events
- Application use case: `ListRecentEvents(n int)` — calls the port, returns last N events
- TUI: root `tea.Model` with one pane, renders the event list
- Composition root: `cmd/clyde/main.go` wires everything
- Tests: unit tests for domain types + use case (table-driven); `teatest` snapshot for the TUI pane
- `golangci-lint` passing with depguard rules

**Pros:**
- Proves ALL layers work together from day one
- Real data visible immediately — motivation to continue
- Architecture validated before feature work begins
- TDD can start at the domain layer (simplest)
- Derisks the "does Bubble Tea v2 work on this machine" question early

**Cons:**
- Must define domain types before having full understanding of all JSONL events
- Some domain types will need evolution in later changes

**Effort:** Medium (2-3 sessions)

---

### Option B: Layer-first

**What:** Build domain types + ports + application use cases with full test coverage. No TUI until layers are battle-tested.

**Pros:**
- Clean domain model without TUI pressure
- Tests are easier to write without teardown concerns

**Cons:**
- No visible output until TUI is added — motivation risk
- May over-engineer domain types for features not yet needed
- The JSONL adapter and domain coupling only becomes visible when TUI is added — surprises late

**Effort:** Medium-High (delayed feedback loop)

---

### Option C: TUI-first

**What:** Design all 7 panes with mock/static data. Wire real adapters only after layout is finalized.

**Pros:**
- Forces layout and UX decisions before any data complexity
- Easy to demo and get feedback on layout

**Cons:**
- Mock data risks building the wrong domain model
- Hexagonal discipline harder to maintain when starting from the UI
- Must throw away mock data adapters — feels wasteful

**Effort:** Medium (but creates throw-away work)

---

### Recommendation: Walking Skeleton (Option A)

Reasoning: The JSONL event shape is now well-understood from sampling (see above). The domain types needed for a 5-event render are small. Starting with a vertical slice validates the entire stack in one change, surfaces Bubble Tea v2 integration issues early, and gives a visible terminal artifact that motivates subsequent changes. Layer-first and TUI-first both delay real feedback and risk building the wrong model. The walking skeleton is the industry-standard approach for this exact scenario — a new greenfield project with a known (if evolving) data source.

## V1 ports

For bootstrap-skeleton specifically, only 2 ports are needed:

| Port | Why needed |
|------|-----------|
| `SessionSource` | Abstracts reading session events from the JSONL files. The use case calls this port; the JSONL adapter implements it. Without this port, the use case would depend directly on the `jsonl` adapter — violating hexagonal discipline and breaking depguard. |
| `Clock` | Abstracts `time.Now()` for computing event age ("3m ago"). Without this, domain/application code has a hidden `time.Now()` dependency that makes tests non-deterministic. A `RealClock` adapter and `FakeClock` test double are trivial to implement. |

Ports deferred to later changes:
- `TodoSource` — reads `~/.claude/todos/*.json`. Not in V1 walking skeleton.
- `GitSource` — reads `.git/` for file change detection. Not in V1.
- `FileWatcher` — wraps fsnotify for live JSONL tailing. V1 can poll with `tea.Tick` without needing this port; introduce in a follow-up change when real-time responsiveness matters.

## Risks

1. **JSONL schema evolves** — Claude Code is actively developed (version 2.1.119 → 2.1.123 observed in a single session). New event types or field renames could break the JSONL adapter. *Mitigation: design the adapter to skip unknown event types gracefully; use `json.RawMessage` for fields that vary. Document the sampled schema version.*

2. **`isSidechain` not used for sub-agents** — Sub-agents are represented as `queue-operation` events, not `isSidechain=true` events as originally assumed. The Subagent panel in V1 will need to parse XML-like `content` from queue-operation events. *Mitigation: defer Subagent panel to a later change; capture this discovery in the spec.*

3. **No cost field in JSONL** — Cost must be computed from token counts + model name + Anthropic's pricing table. The pricing table changes. *Mitigation: implement cost computation as a pure domain function parameterized on a price-per-token config struct; the config can be updated without structural changes.*

4. **Usage is per-assistant-event, not per-turn** — Aggregating usage per "turn" or per "session" requires accumulating across all assistant events. *Mitigation: `Usage.Add(other Usage) Usage` method on the domain type; the use case accumulates.*

5. **Bubble Tea v2 API changes from v1** — The docs sampled are primarily v1 (v1.3.10). v2 may have API differences. *Mitigation: check `go.sum` / `go.mod` pinned version when wiring. The core MVU interface and Cmd/Batch/Tick patterns are stable across versions.*

6. **golangci-lint depguard blocks `os` in domain** — The `.golangci.yml` denies `os` in `internal/domain/**`. Domain types cannot read files directly. *Mitigation: this is intentional and correct — all I/O goes through ports. Just don't let domain types call `os.Open`. Keep domain as pure value objects.*

7. **No `go.mod` yet** — The project has no Go source files. `go mod init` needs to run before any other Go tooling. *Mitigation: first task in bootstrap-skeleton is always `go mod init github.com/clyde-tui/clyde`.*
