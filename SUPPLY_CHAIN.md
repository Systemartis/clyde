# Supply chain

This document describes how clyde release artifacts are produced, what guarantees come with them, and how to verify a binary you've downloaded.

## What every release ships

Each tag (`v*`) triggers `.github/workflows/release.yml`, which runs `goreleaser release --clean` on a clean GitHub-hosted runner. The release uploads:

| Artifact | Format | Provenance |
|----------|--------|-----------|
| `clyde_<version>_<os>_<arch>.tar.gz` | tarball with the `clyde` binary, `README.md`, `LICENSE`, `CONTRIBUTING.md` | Built from the tagged commit; `-trimpath` + stripped symbols |
| `<archive>.spdx.json` | SPDX 2.3 JSON | Generated by [`syft`](https://github.com/anchore/syft) over each archive |
| `checksums.txt` | one SHA-256 per archive (and SBOM) | Computed inside goreleaser |
| `checksums.txt.sig` | cosign blob signature | Signed in keyless mode by [`cosign`](https://github.com/sigstore/cosign) using GitHub's OIDC identity |
| `checksums.txt.pem` | Fulcio-issued certificate | Public certificate proving the OIDC identity at signing time |
| build-provenance attestation | in-toto SLSA Provenance v1 | Generated by `actions/attest-build-provenance` and stored in the repo's `/attestations` endpoint. Verifiable with `gh attestation verify` |

Provenance: The signing identity is the GitHub Actions workflow `Systemartis/clyde/.github/workflows/release.yml` running on a `refs/tags/v*` ref. The cosign certificate's `subject-alternative-name` and `issuer` are pinned to that identity in the verify recipe below — a tampered release won't pass.

## Verifying a downloaded binary

You need [`cosign`](https://github.com/sigstore/cosign#installation) (≥ v2.2) and `sha256sum` / `shasum -a 256`.

```sh
VERSION=0.1.0
OS=$(uname -s | tr '[:upper:]' '[:lower:]')
ARCH=$(uname -m | sed 's/x86_64/amd64/;s/aarch64/arm64/')
BASE="https://github.com/Systemartis/clyde/releases/download/v${VERSION}"

# 1. Download the archive + checksum manifest + signature.
curl -fsSLO "${BASE}/clyde_${VERSION}_${OS}_${ARCH}.tar.gz"
curl -fsSLO "${BASE}/checksums.txt"
curl -fsSLO "${BASE}/checksums.txt.sig"
curl -fsSLO "${BASE}/checksums.txt.pem"

# 2. Verify the checksum manifest's signature came from our release workflow.
cosign verify-blob \
  --certificate checksums.txt.pem \
  --signature checksums.txt.sig \
  --certificate-identity-regexp "https://github.com/Systemartis/clyde/.github/workflows/release.yml@refs/tags/v.*" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  checksums.txt

# 3. Verify the archive against the manifest.
sha256sum -c checksums.txt --ignore-missing
```

If step 2 fails, **stop**. The archive may match the manifest's hash (step 3 may pass) but a manifest you can't authenticate is worthless — anyone could have replaced both.

### Verifying via SLSA attestation (alternative to cosign)

If you'd rather not install `cosign`, the same archive is covered by a GitHub-stored SLSA attestation. Verify with the `gh` CLI:

```sh
gh attestation verify "clyde_${VERSION}_${OS}_${ARCH}.tar.gz" --owner Systemartis
```

This validates the in-toto provenance: that this exact byte sequence was produced by the workflow `Systemartis/clyde/.github/workflows/release.yml` running on a `refs/tags/v*` ref.

## SBOM consumption

```sh
# Feed the SBOM into grype to scan the archive's binary against the
# OSS Vulnerability Database without re-deriving the dep tree.
grype sbom:./clyde_${VERSION}_${OS}_${ARCH}.tar.gz.spdx.json

# Or trivy, if you prefer:
trivy sbom ./clyde_${VERSION}_${OS}_${ARCH}.tar.gz.spdx.json
```

The SBOM lists every Go module the binary embeds plus their licenses and known vulnerability identifiers (CVE / GHSA). New advisories landing after the release will surface here even if they didn't exist when we tagged.

## Build reproducibility

The release builds with `-trimpath` and `-ldflags "-s -w -X .../version.{version,commit,date}=..."`. Two things keep them reproducible:

- The Go toolchain version is pinned via `actions/setup-go@... { go-version: '1.26' }` in the release workflow.
- All third-party GitHub Actions are pinned to commit SHAs (see `.github/workflows/*.yml`).

What we do **not** currently provide:

- **SLSA Level 3 provenance attestation** — under consideration; would require switching to the slsa-framework generator action.
- **Reproducible byte-for-byte builds across runners.** The Go compiler's `-buildid` and runtime metadata can vary between toolchain patch releases. We pin the major (`1.26`) but not the patch.

## Dependency hygiene

- `go.mod` pins direct dependencies; `go.sum` records the hash of every transitive dep.
- `govulncheck v1.3.0` runs on every PR (`.github/workflows/ci.yml`) and on `main`.
- CodeQL `security-and-quality` query suite runs weekly + on every PR (gated to public repos — runs once we flip visibility).
- Dependabot (`.github/dependabot.yml`) auto-opens PRs for both `gomod` and `github-actions` ecosystems on a weekly cadence.

## Reporting a supply-chain issue

See [SECURITY.md](./SECURITY.md). Use the same private channel — supply-chain issues (a maliciously-bumped action SHA, a replaced binary on a release page, a typosquatted dep) are in scope.
